# قواعد نگارشی و فنی فارسی

این فایل house style موهتیوس است. پایه‌ی آن دستور خط فارسی است، اما در مواردی که خوانایی دیجیتال و صدای نویسنده مهم‌تر است، تصمیم صریح و یکدست ثبت شده است.

## نویسه‌های فارسی

همیشه از نویسه‌های فارسی استفاده کن:

| نادرست | درست |
|---|---|
| ي | ی |
| ك | ک |
| ة در واژه‌ی فارسی | ه یا ت، بر اساس واژه |
| شکل‌های ناسازگار اضافه | house style ثابت |

وجود `ي` و `ك` در متن فارسی خطای lint است، مگر داخل code block، URL یا نقل‌قول دقیق.

## نیم‌فاصله

از U+200C ZERO WIDTH NON-JOINER استفاده کن:

- می‌رود
- نمی‌شود
- نوشته‌ها
- پروژه‌ها
- نسخه‌ی
- حرفه‌ای
- به‌روز
- هم‌زمان

نیم‌فاصله را برای زیباترکردن هر ترکیب دلخواه استفاده نکن. اگر دو واژه از نظر معنایی مستقل‌اند، فاصله‌ی کامل لازم است.

## تصمیم house style برای «ه‌ی»

در محتوای موهتیوس از ساختار خوانای دیجیتال `ه‌ی` استفاده می‌شود:

- نسخه‌ی اول
- درباره‌ی پروژه
- تجربه‌ی کار
- خانه‌ی شخصی

این انتخاب عمدی است و باید در یک متن ثابت بماند. Agent نباید آن را به شکل‌های ناسازگار تبدیل کند.

## نشانه‌گذاری

در متن فارسی:

- ویرگول: `،`
- نقطه‌ویرگول: `؛`
- پرسش: `؟`
- نقل‌قول: `«...»`
- دونقطه: `:`
- پرانتز: `(…)`
- خط تیره‌ی معترضه: `—`

قاعده:

- پیش از نشانه فاصله نگذار.
- پس از نشانه یک فاصله بگذار، مگر پایان خط یا پیش از پرانتز بسته.
- چند علامت تعجب پشت سر هم ممنوع است.
- علامت تعجب در مقاله‌ی فنی تقریباً همیشه غیرضروری است.

## اعداد

- در نثر فارسی، شمارش عمومی را با رقم فارسی بنویس: `۳ بخش`، `۳۰ روز`.
- version، identifier، protocol، مدل سخت‌افزار و code را تغییر نده: `V1`، `ESP8266`، `HTTP/2`، `SHA-256`.
- تاریخ frontmatter و ISO را تغییر نده: `2026-07-10`.
- اعداد داخل code، command، URL، filename و schema دست‌نخورده بمانند.
- اگر واحد بخشی از syntax فنی است، format اصلی را حفظ کن: `64 KB`.

## واژه‌های انگلیسی در جمله‌ی فارسی

اصل:

> اصطلاح فنی شناخته‌شده را طبیعی نگه دار؛ برای فارسی‌نمایی، ترجمه‌ی ساختگی تولید نکن.

- نام محصول، API، protocol و primitive را به انگلیسی نگه دار.
- واژه‌ی انگلیسی را فقط برای مدرن‌به‌نظررسیدن وارد جمله نکن.
- اولین اشاره می‌تواند توضیح فارسی کوتاه داشته باشد.
- یک اصطلاح را در یک مقاله با چند معادل جابه‌جا نکن.

مثال:

> پیام ابتدا وارد Queue می‌شود تا پردازش webhook از کارهای پس‌زمینه جدا بماند.

## جهت متن و Markdown

- code block، inline code، command، URL و identifier را LTR نگه دار.
- برای حل bidi، متن code را با نویسه‌های نامرئی تصادفی دست‌کاری نکن.
- نام فایل و مسیر را داخل backtick قرار بده.
- code fence و زبان آن را حفظ کن.
- frontmatter را بدون درخواست صریح تغییر نده.
- link targetها را تغییر نده.
- anchor headingها را بدون بررسی لینک داخلی عوض نکن.
- جدول را فقط برای مقایسه‌ی واقعی استفاده کن.
- list را برای شکستن نثر عادی به bulletهای مصنوعی به‌کار نبر.

## عنوان‌ها

عنوان خوب:

- مسئله یا تصمیم مشخصی را معرفی می‌کند.
- کوتاه و قابل‌فهم است.
- clickbait نیست.
- در پایان نقطه ندارد.
- از «همه‌چیز درباره‌ی...» دوری می‌کند.

نمونه:

- «چرا routing نکونیموس را از payload جدا کردم»
- «ساخت یک inbox کوچک با Cloudflare Email Workers»
- «نسخه‌ی اول چه چیزی را عمداً حل نمی‌کند»

## پاراگراف

- هر پاراگراف یک ایده‌ی مرکزی داشته باشد.
- پاراگراف تک‌جمله‌ای فقط برای تأکید یا transition باشد.
- پاراگراف بلند را در نقطه‌ی تغییر reasoning بشکن.
- شروع چند پاراگراف پشت سر هم با یک عبارت ثابت ممنوع است.
- heading برای هر دو پاراگراف متن را بیش از حد خرد می‌کند.

## فهرست‌ها

فهرست مناسب است وقتی ترتیب عملیات، مقایسه، checklist یا constraint scanپذیر لازم است.

فهرست نامناسب است وقتی:

- هر bullet فقط یک جمله‌ی عادی از یک پاراگراف است.
- سه مزیت کلی برای تبلیغ پروژه ساخته شده است.
- list جای reasoning را گرفته است.

## زمان و وضعیت

- feature موجود: زمان حال روشن
- کار تمام‌شده: گذشته
- roadmap: آینده یا «قرار است»
- ایده‌ی احتمالی: «ممکن است»، «می‌توان»
- تصمیم قطعی‌نشده: با فعل قطعی ننویس

نادرست:

> در نسخه‌ی بعد پرداخت اضافه می‌شود.

درست، اگر تصمیم هنوز قطعی نیست:

> پرداخت یکی از گزینه‌های نسخه‌ی بعد است، اما هنوز بخشی از V1 نیست.

## ویرایش کم‌خطر

- frontmatter را preserve کن.
- component، shortcode و directiveهای Markdown را تغییر نده.
- code و config را ترجمه نکن.
- quote دقیق را بدون درخواست paraphrase نکن.
- link شکسته تولید نکن.
- encoding را UTF-8 نگه دار.
