# الگوهای ساختاری

این‌ها recipe هستند، نه قالب اجباری.

## مسئله → constraint → تصمیم

```text
مسئله چه بود؟
چه چیزی انتخاب‌ها را محدود می‌کرد؟
چه گزینه‌هایی وجود داشت؟
چرا این تصمیم گرفته شد؟
چه چیزی در ازای آن از دست رفت؟
نتیجه‌ی فعلی چیست؟
```

## چیزی که ساده به‌نظر می‌رسید

```text
در ظاهر، کار فقط X بود.
در عمل، Y و Z مسئله را پیچیده کردند.
راه‌حل اولیه A بود، اما در B شکست خورد.
برای همین طراحی به C تغییر کرد.
```

## بازطراحی

```text
نسخه‌ی قبلی چه فرضی داشت؟
کدام رفتار واقعی آن فرض را نقض کرد؟
چه بخش‌هایی حذف یا ساده شدند؟
مدل جدید چه invariantی دارد؟
چه ریسکی هنوز باقی مانده است؟
```

## معرفی پروژه

```text
پروژه در یک جمله
مسئله‌ی مبنا
تجربه‌ی اصلی کاربر
سه تصمیم مهم، نه سه feature تبلیغاتی
وضعیت و مرز نسخه
```

## تصمیم معماری

```text
تصمیم:
دلیل:
گزینه‌های ردشده:
trade-off:
شرط بازنگری:
```

می‌توان این ساختار را به نثر تبدیل کرد.

## شروع‌های مناسب

- مسئله‌ی واقعی
- تصمیم غیرمنتظره
- شکاف میان انتظار و عمل
- مرز نسخه
- سؤال مشخصی که پروژه به آن پاسخ می‌دهد

## شروع‌های ضعیف

- تعریف واژه از فرهنگ لغت
- تاریخچه‌ی کلی فناوری
- ادعای اینکه موضوع روزبه‌روز مهم‌تر می‌شود
- مقدمه‌ای قابل‌استفاده برای هر پروژه
- hook هیجانی بی‌ارتباط

## پایان‌های مناسب

- وضعیت فعلی و چیزی که باز است
- شرطی که تصمیم را در آینده تغییر می‌دهد
- insight قابل‌انتقال
- لینک code، demo یا docs
- قدم بعدی واقعی

## پایان‌های ضعیف

- تکرار تمام مقاله
- «در نهایت می‌توان گفت...»
- دعوت کلی به embraceکردن فناوری
- وعده‌ی مبهم
- CTA فروش‌محور
