# modeهای محتوایی

قبل از نوشتن یا بازنویسی، یک mode اصلی انتخاب کن. mode فرعی ممکن است وجود داشته باشد، اما ساختار باید یک هسته‌ی روشن داشته باشد.

## ۱. Project Page

### سؤال مخاطب

این پروژه چیست، چرا وجود دارد، چه چیزی ساخته شده و وضعیت فعلی آن کجاست؟

### ساختار پیشنهادی

1. تعریف روشن در ۱ تا ۲ پاراگراف
2. مسئله‌ی آغازین
3. تجربه یا قابلیت اصلی
4. معماری در سطح لازم برای معرفی
5. تصمیم‌های شاخص و دلیل آن‌ها
6. وضعیت فعلی و مرز V1
7. لینک‌ها یا قدم بعدی

### لحن

- مطمئن، اما نه تبلیغاتی
- فشرده‌تر از مقاله
- محصول‌محور، نه changelogمحور
- بدون dumpکردن همه‌ی معماری

### ضدالگو

- تاریخچه‌ی عمومی صنعت
- فهرست تکنولوژی بدون دلیل
- roadmap به‌عنوان feature فعلی
- CTA بازاری

## ۲. Lab Note

### سؤال مخاطب

چه چیزی امتحان شد، چه چیزی جواب نداد و از آن چه یاد گرفته شد؟

### ساختار پیشنهادی

1. context کوتاه
2. فرض اولیه
3. چیزی که در عمل رخ داد
4. تصمیم یا اصلاح
5. نتیجه‌ی فعلی
6. پرسش یا قدم باز

### لحن

- شخصی‌تر
- مجاز به بیان تردید و تغییر نظر
- جزئیات فنی انتخابی
- پایان می‌تواند باز باشد

### ضدالگو

- تبدیل شکست کوچک به داستان قهرمانانه
- نتیجه‌گیری کلی درباره‌ی کل صنعت
- پنهان‌کردن بخش حل‌نشده

## ۳. Technical Article

### سؤال مخاطب

این مسئله چطور تحلیل و حل شد و چه چیزی از آن قابل‌انتقال است؟

### ساختار پیشنهادی

1. مسئله و context واقعی
2. constraints
3. گزینه‌های بررسی‌شده
4. تصمیم معماری و trade-off
5. جزئیات پیاده‌سازی
6. failure mode یا محدودیت
7. نتیجه و کاربرد عمومی‌تر

### لحن

- دقیق و آموزشی
- reasoningمحور
- code فقط جایی که توضیح را بهتر می‌کند
- نه documentation خام و نه روایت صرف

### ضدالگو

- tutorial بدون چرایی
- معماری بدون constraint
- performance claim بدون measurement
- نتیجه‌ای که مقدمه را تکرار می‌کند

## ۴. Release Note / Announcement

### سؤال مخاطب

چه چیزی منتشر شده، اکنون چه کاری انجام می‌دهد و مرز نسخه چیست؟

### ساختار پیشنهادی

1. یک جمله‌ی روشن درباره‌ی انتشار
2. ۳ تا ۵ تغییر مهم
3. یک نکته درباره‌ی طراحی
4. محدودیت یا خارج از scope
5. لینک و قدم بعدی

### لحن

- کوتاه
- مشخص
- بدون هیجان مصنوعی
- همراه با status دقیق

### ضدالگو

- «با افتخار اعلام می‌کنیم»
- ۱۰ bullet هم‌سطح
- صفت بزرگ بدون evidence
- پنهان‌کردن beta یا manual-test

## ۵. Personal Engineering Essay

### سؤال مخاطب

این تجربه چه تغییری در نحوه‌ی فکرکردن یا ساختن ایجاد کرد؟

### ساختار پیشنهادی

1. صحنه یا مسئله‌ی مشخص
2. برداشت اولیه
3. برخورد با واقعیت
4. تغییر نگاه
5. نتیجه‌ی شخصی و فنی

### لحن

- شخصی‌تر و آرام
- همچنان مشخص و grounded
- بدون تبدیل‌شدن به متن انگیزشی

## ۶. Persian Counterpart of English Content

این mode ترجمه‌ی خط‌به‌خط نیست.

### workflow

1. factual outline انگلیسی را استخراج کن.
2. claims، links، code و status را freeze کن.
3. ساختار فارسی را مستقل طراحی کن.
4. idiom انگلیسی را لفظ‌به‌لفظ ترجمه نکن.
5. parity را در سطح معنا نگه دار، نه تعداد جمله.

### ممنوع

- ترجمه‌ی مستقیم عبارت بازاری
- معادل فارسی عجیب برای APIها
- افزودن claim جدید
- حذف limitation برای روان‌ترشدن

## ۷. Social Derivative

پست LinkedIn، X یا معرفی کوتاه باید از مقاله مشتق شود، نه نسخه‌ی فشرده‌ی مکانیکی آن.

### ساختار

1. hook واقعی از مسئله
2. یک insight یا تصمیم
3. نتیجه‌ی مشخص
4. link یا دعوت ساده

### لحن

- کوتاه و طبیعی
- بدون emoji اجباری
- بدون engagement bait
