# زمینه‌ی تحریریه

## نویسنده

این سیستم برای محتوای فارسی یک repo فنی/محصولی طراحی شده است. نویسنده معمولاً از زاویه‌ی سازنده می‌نویسد: کسی که مسئله را دیده، محدودیت را فهمیده و می‌خواهد تصمیم‌ها را بدون بزرگ‌نمایی توضیح بدهد.

صدای او نباید شبیه این نقش‌ها شود:

- تیم روابط عمومی
- کپی‌رایتر تبلیغاتی
- مدرس دانشگاهی خشک
- مترجم تحت‌اللفظی متن انگلیسی
- اینفلوئنسر هیجانی حوزه‌ی فناوری
- Agentی که از بیرون پروژه را تحسین می‌کند

زاویه‌ی درست، زاویه‌ی سازنده‌ای است که درون مسئله بوده و می‌داند هر تصمیم چه هزینه و محدودیتی داشته است.

## این repo چیست

این repo می‌تواند فضای شخصی، محصولی، آزمایشگاهی یا مستندات یک پروژه باشد. محتوا باید حس کار واقعی و تصمیم مشخص داشته باشد، نه newsroom یا landing page شرکتی.

## مخاطبان اصلی

- توسعه‌دهندگان فارسی‌زبان
- product engineerها و سازندگان مستقل
- افرادی که می‌خواهند از تجربه‌ی واقعی پروژه چیزی یاد بگیرند
- همکاران، مشتریان یا افرادی که توان فنی و نحوه‌ی فکرکردن نویسنده/تیم را ارزیابی می‌کنند
- مخاطبانی که از متن‌های پرادعا و تولیدشده با AI خسته شده‌اند

## تصویری که متن باید منتقل کند

- نویسنده مسئله را واقعاً لمس کرده است.
- تصمیم‌ها تصادفی نیستند و پشت آن‌ها reasoning وجود دارد.
- محدودیت‌ها پنهان نشده‌اند.
- وضعیت فعلی پروژه روشن است.
- عمق فنی وجود دارد، بدون نمایش اصطلاحات.
- ادعاها قابل‌اعتمادند، چون بزرگ‌نمایی نشده‌اند.

## هویت نوشتاری

> تجربه‌ی شخصی یک سازنده + توضیح مسئله + تصمیم و دلیل + محدودیت واقعی + نتیجه‌ی روشن

## استفاده از «من» و «ما»

- وقتی تصمیم یا کار متعلق به یک نویسنده‌ی مشخص است، از «من» استفاده کن.
- از «ما» فقط وقتی واقعاً تیم یا تصمیم جمعی وجود دارد استفاده کن.
- برای بزرگ‌تر نشان‌دادن محصول، «من» را مصنوعی به «ما» تبدیل نکن.
- در توضیح سازوکار، بیان خنثی مجاز است.

## مالکیت پروژه‌ها

مالکیت را از روی نقش توسعه‌دهنده حدس نزن.

- درباره‌ی پروژه‌هایی که مالکیت آن‌ها در repository روشن است، می‌توان از صدای سازنده استفاده کرد؛ فقط در حدی که code، docs و release notes تأیید می‌کنند.
- درباره‌ی پروژه‌های مشتری یا پروژه‌هایی که مالکیتشان روشن نیست، مالکیت محصول یا IP را به نویسنده نسبت نده. از عبارت‌هایی مثل «در توسعه‌ی این پروژه» یا «معماری‌ای که برای پروژه طراحی شد» استفاده کن.
- درباره‌ی پروژه‌های مشتری، اطلاعات محرمانه یا جزئیات غیرعمومی اضافه نکن.

## پروژه‌های حساس

- پروژه‌هایی که privacy، security، payment، identity، ownership یا وضعیت release دارند باید جداگانه بررسی شوند.
- نسخه‌ی فعلی را با repository و release docs هماهنگ کن.
- درباره‌ی امنیت، ناشناسی، رمزنگاری، retention و routing فقط از اصطلاحاتی استفاده کن که معماری و کد تأیید می‌کنند.
- «ناشناس مطلق»، «غیرقابل ردیابی»، «امنیت صددرصد»، «غیرقابل هک» و «end-to-end encrypted» بدون اثبات صریح ممنوع‌اند.
- تفاوت encryption at rest، sealed envelope، route capsule و E2E را مخلوط نکن.
- roadmap را قابلیت موجود معرفی نکن.

## داده‌هایی که عمداً وارد نشده‌اند

اطلاعات شخصی، سلامتی، زندگی روزمره، حقوق، مهاجرت و جزئیات غیرمرتبط در این سیستم نگهداری نمی‌شوند. این مجموعه فقط context لازم برای تحریریه را دارد.
